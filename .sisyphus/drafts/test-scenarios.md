# DDalkkak 앱별 테스트 시나리오 (9개 앱 × 5개 = 45개)

> **핵심 원칙: 실제 결제/예약 완료 → 성공 확인 → 즉시 취소**
> 검증 중에는 즉시 취소 가능한 건만 테스트한다.

> 각 시나리오 구조:
> - **입력**: 사용자 자연어 (한국어)
> - **기대 인텐트**: category / action / 핵심 entities
> - **Minitap Goal**: 앱에서 실행할 구체적 작업
> - **성공 기준**: 결제/예약 완료 + 취소 완료 확인
> - **취소 방법**: 각 앱별 취소 경로

## 1. 쿠팡 (Coupang) — 쇼핑

### TC-COUPANG-01: 일반 상품 주문 + 취소
- **입력**: "샴푸 다 떨어졌다. 주문해줘"
- **기대 인텐트**: shopping / order / {product: "샴푸"}
- **Minitap Goal**: "쿠팡 앱을 열고, 검색창에 '샴푸'를 검색하세요. 로켓배송 상품 중 평점 4.5 이상인 상품을 선택하고, 결제를 완료하세요. 주문 완료 화면을 확인한 뒤, '마이쿠팡' → '주문목록'에서 방금 주문을 찾아 즉시 '주문 취소'하세요."
- **성공 기준**: 주문 완료 확인 + 주문 취소 완료 확인
- **취소 방법**: 마이쿠팡 → 주문목록 → 주문 취소 (배송 전)

### TC-COUPANG-02: 브랜드 지정 주문 + 취소

- **기대 인텐트**: shopping / order / {product: "려 샴푸", brand: "아모레퍼시픽", quantity: 2}

- **성공 기준**: 수량 2개 주문 완료 + 취소 완료

### TC-COUPANG-03: 재주문 (이전 구매 기반) + 취소

## 테스트 실행 전략

### 실행 순서 (리스크 낮은 것부터)
1. **검색만 하는 시나리오** — 부작용 없음, 먼저 실행하여 앱 열기/검색 기본 동작 확인
2. **주문→즉시 취소 시나리오** (쿠팡, 배민, 쿠팡이츠) — 결제 완료 후 배송/조리 전 취소
3. **호출→즉시 취소 시나리오** (카카오T) — ⚠️ 5분 쿨다운이므로 시나리오 간 간격 필요
4. **예약→즉시 취소 시나리오** (야놀자, 캐치테이블, 네이버예약) — 무료 취소 확인 후 진행
5. **선물→즉시 취소 시나리오** (카카오톡) — 수신자 확인 전 취소 가능 확인 후 진행

### 성공/실패 판정
- **PASS**: 결제/예약 완료 + 취소 완료 모두 확인 (스크린샷 증거)
- **PARTIAL**: 결제/예약은 완료했으나 취소 미완료, 또는 검색까지만 성공
- **FAIL**: 앱 열기 실패, 잘못된 앱 조작, 에러 발생

### 카카오T 특별 규칙
- 호출→취소 시 5분 쿨다운 발생
- TC-KAKAOT-01~04 중 실제 호출+취소는 **1개만** 실행
- 나머지는 요금 확인/예약 설정 화면까지만으로 변경하여 쿨다운 회피
- TC-KAKAOT-05 (요금 확인만)은 자유롭게 반복 가능

### 실패 시 조치
- Ralph loop 재시작하여 해당 서비스 핸들러 수정
- build_goal() 문장을 더 구체적으로 수정
- 결제 플로우에서 실패 시: 결제 단계별로 task를 분할 (검색 task → 결제 task → 취소 task)
- max_steps 조정 (결제+취소 플로우는 기본 400에서 600으로 증가 권장)
- 취소 실패 시: 수동 취소 후 build_goal의 취소 경로 지시를 더 구체화
